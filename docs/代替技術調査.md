# Markmiru 代替技術調査

最終更新: 2026-06-28

> **2026-06-28 改訂（Go 中心化）**: [`技術選定.md`](技術選定.md) の方針転換（Go 中心・サーバサイド描画）に伴い、本書も Go 側ライブラリの比較へ全面改訂した。旧構成（Svelte/markdown-it/highlight.js/DOMPurify/CodeMirror/Vite）の比較は git 履歴を参照。

本アプリの要件（**軽量・高速起動・綺麗な閲覧・GFM＋mermaid・閲覧重視／「ほぼ Go」**）に照らし、各領域で採用技術と代替候補を比較する。

## 0. 結論サマリ

| 領域 | 採用 | 主な代替 | 判定 |
|------|------|----------|------|
| UI 構成 | **Go-SSR ＋ htmx** | Svelte/TS（旧）/ Go→WASM / 純 Go GUI | **Go-SSR 採用**（MUST 維持で最大限 Go 化。[`技術選定.md`](技術選定.md) §5.1） |
| Markdown 解析 | **goldmark** | gomarkdown / blackfriday / golang-commonmark | **goldmark 維持**（GFM・脚注・拡張性・保守性） |
| 構文ハイライト | **chroma** | （JS の highlight.js/Shiki は不採用） | **chroma 採用**（純 Go・閲覧/編集で共用） |
| HTML サニタイズ | **bluemonday** | （JS の DOMPurify は不採用） | **bluemonday 採用**（純 Go・allowlist） |
| 図表 | **mermaid.js** | mmdr（Rust・将来）/ 純 Go（無し） | **mermaid.js 維持**（純 Go 代替なし。§5） |
| 編集エディタ | **textarea オーバーレイ＋chroma** | CodeMirror 6 / CodeJar / Monaco | **オーバーレイ採用**（ハイライト本体を Go 化） |

## 1. UI 構成：Go-SSR＋htmx vs 代替

「ほぼ Go」化の3案（A: Go-SSR＋htmx ／ B: Go→WASM ／ C: 純 Go GUI）の詳細比較とコード量見積りは [`技術選定.md`](技術選定.md) §5.1 に集約した。要点のみ:

- **A（採用）**: Wails v2 ＋ Go が `html/template` で HTML を生成し、htmx で操作をインタラクティブ化。**MUST（高速起動・綺麗な閲覧・mermaid）を全て維持**しつつ、自作 JS を数百行のグルーまで圧縮できる。Go 率は約 62〜84%。
- **B（不採用）**: Go→WASM（go-app/vugu/vecty）。Go 率は最大化されるが WASM 数 MB で**高速起動 MUST と衝突**、フレームワークが実験段階。
- **C（不採用）**: 純 Go GUI（Fyne/Gio）。JS ゼロだが **Markdown 描画品質が低く MUST に抵触**、PDF も困難。

> 旧構成（Svelte/TS）は軽量・高速で UI 自体は優秀だったが、アプリロジックの大半が TS 側に集まり「Go で書かれたアプリ」と言い難くなる根本要因だった。Go-SSR はこの構造を反転させる。

## 2. Markdown 解析：goldmark vs 代替（Go）

| 候補 | 特徴 | 本件評価 |
|------|------|----------|
| **goldmark（採用）** | CommonMark 完全準拠。GFM（表・打消し線・自動リンク・タスクリスト）・脚注・定義リストを拡張で追加。AST が綺麗で拡張容易。Hugo 採用で実績豊富 | **採用**。markdown-it の機能を過不足なく代替し、将来の数式等の拡張余地も大 |
| gomarkdown / blackfriday | 老舗。blackfriday v2 の系譜。GFM 相当は可能だが拡張性・準拠度で goldmark に劣る | 代替候補だが優位性なし |
| golang-commonmark | markdown-it の Go 移植。準拠度は高いが拡張エコシステムが小さい | 移植版のため将来性に不安 |

- mermaid は goldmark のコードブロック（言語名 `mermaid`）を `<pre class="mermaid">` として出力し、WebView 側の mermaid.js が描画する（goldmark-mermaid 拡張の client-side モード相当を自前実装または利用）。

---

## 3. 構文ハイライト：chroma（純 Go）

| 候補 | 特徴 | 本件評価 |
|------|------|----------|
| **chroma（採用）** | 純 Go。Pygments 由来の多言語レキサ。色付け HTML（インライン/クラス）を出力 | **採用**。閲覧時の色付けと、編集オーバーレイの色付けを**同一エンジンで共用**できる |
| highlight.js / Shiki / Prism（JS） | 高品質だが JS。Go-SSR では不要 | **不採用**（JS を増やさない方針） |

- 「綺麗に表示（MUST）」の観点では Shiki（VS Code 同等品質）が JS 側の最有力だったが、Go 中心化により**ハイライトも Go（chroma）へ寄せる**。chroma のテーマで十分な品質が得られるかは実装時に確認し、不足ならスタイル（CSS）で補正する。

---

## 4. HTML サニタイズ：bluemonday（純 Go）

| 候補 | 特徴 | 本件評価 |
|------|------|----------|
| **bluemonday（採用）** | 純 Go・allowlist 方式。Go 標準 `net/html` ベースの高速トークナイザ | **採用**。HTML を Go 側で生成・無害化してから WebView へ渡す構成に最適 |
| DOMPurify（JS） | ブラウザ DOM ベースで堅牢だが JS。クライアントで実行 | **不採用**（描画を Go-SSR 化したため不要） |

- 方針: Markdown→HTML（goldmark）→ サニタイズ（bluemonday）→ WebView 表示、を**全て Go プロセス内**で完結させる。外部リンクへの `rel` 付与・外部画像の遮断制御も bluemonday のポリシーで表現する。

---

## 5. 図表：mermaid（純 Go 代替なし）

- 要件が「mermaid を利用した図表」であり、**純 Go で mermaid を描画する手段は存在しない**（純 Go の go-mermaid 等はパーサ/検証のみ）。Go 内 JS エンジン（goja/v8go）も DOM 非対応で mermaid.js を走らせられない。
- → **mermaid.js を WebView 内で使用**（現行方式を維持）。これが「ほぼ Go」で消せない唯一の JS。
- **将来オプション（情報のみ・未確定）**: **mmdr（mermaid-rs-renderer、Rust）** は mermaid をブラウザ/Node 無しで SVG 描画でき、採用すれば mermaid.js を排除できる。ただし Rust サイドカー依存・v0.2 系の成熟度・全図種カバレッジ未検証のため**現時点では採用しない**。詳細は [`技術選定.md`](技術選定.md) §6。

## 6. 編集エディタ：textarea オーバーレイ＋chroma vs 代替

| 候補 | バンドル/JS 量 | 特徴 | 本件評価 |
|------|----------------|------|----------|
| **textarea オーバーレイ＋chroma（採用）** | 数十行のグルー JS | 透明 `<textarea>` を chroma(Go) 出力の色付け層に重ねる。**ハイライト本体は Go** | **採用**。編集時ハイライトを保ちつつ JS を最小化。閲覧重視の本アプリに十分 |
| CodeJar / LDT / react-simple-code-editor | 極小ライブラリ | 同じオーバーレイ手法を提供。ハイライトは JS 側 | 採用案と同手法だが、色付けを JS に持つ分 Go 率が下がる |
| CodeMirror 6（旧採用） | 約124KB＋設定 | 行番号・高機能・高カスタマイズ | **不採用**。JS の大きな塊で「ほぼ Go」に反する。閲覧重視のため過剰 |
| Monaco / Ace | 2MB 超 / 約98KB | VS Code エンジン等 | 過剰・重い |

- トレードオフ: 行番号・矩形選択・補完等の高機能は失う。本アプリは**閲覧重視・編集は軽微な修正用途（SHOULD）**のため許容する。

---

## 7. フロントのツールチェーン：ビルド廃止

- Go-SSR では UI を Go が生成するため、**Svelte コンパイル・Vite バンドル・npm 依存解決が不要**になる。
- 残る JS（**mermaid.js の配布物・htmx の単一ファイル・自作グルー数百行**）は、ソースに直接 vendor し `//go:embed` で実行バイナリへ同梱すればよい（バンドラ不要）。
- これにより **Node.js / npm / Vite / Svelte / 関連の固定バージョン管理（volta・engine-strict・ビルドスクリプトの node/npm 検査）が一括で不要**になり、ビルドは「Go ＋ Wails」のみに単純化される。
- 旧構成（Node＋Vite＋Svelte）との対比は git 履歴を参照。

---

## 8. 総括

- **UI=Go-SSR＋htmx / Markdown=goldmark / ハイライト=chroma / サニタイズ=bluemonday / 編集=textarea オーバーレイ＋chroma** により、描画・サニタイズ・ハイライト・アプリロジックを**全て Go に集約**できる。
- **JS は mermaid.js（純 Go 代替なし）＋ htmx ＋ 数百行のグルー**に限定され、Svelte/TS の自作コードと Node/Vite ビルドは廃止される。
- **mermaid だけは将来も JS が残る前提**（mmdr による排除は未確定の将来オプション）。これが「ほぼ Go」の現実的な到達点。
