# Markmiru 技術選定

最終更新: 2026-06-28

> **2026-06-28 改訂（Go 中心化）**: 当初は Wails v2 ＋ Svelte/TypeScript で実装したが、「機能の大半が Svelte/TS にあり、Go で書かれたアプリとは言い難い」状態を解消するため技術構成を再検討し、**Go 中心（サーバサイド描画）構成**へ方針転換した。本書は再検討後の結論を正とする。旧構成（Svelte/TS）の詳細は git 履歴を参照。

## 1. アプリ概要

Markdown ドキュメントの**閲覧・編集**を行うデスクトップアプリケーション。

- 綺麗にレンダリングして表示する「**閲覧モード**」と、Markdown 表記を直接編集する「**編集モード**」を切り替えて使う
- どちらかといえば**閲覧に重点**を置く

## 2. 要件と優先度

要件は従来から**変更なし**。

| 区分 | 要件 |
|------|------|
| MUST | Markdown を綺麗にレンダリングして閲覧できる |
| MUST | mermaid を利用した図表もレンダリングして閲覧できる |
| MUST | プラグインなしの単体アプリとして動作する |
| MUST | 起動が素早い |
| MUST | マルチタブで複数ドキュメントを切り替えて表示できる |
| SHOULD | スタイルの修正/変更ができる |
| SHOULD | PDF 出力できる |
| SHOULD | クロスプラットフォーム対応 |
| SHOULD | Markdown の編集ができる（編集モードでの編集） |
| SHOULD | 「閲覧モード」と「編集モード」を切り替えられる |

**追加方針（実装言語）**: 上記要件を満たしたうえで、実装を**「ほぼ Go」**で行う。自作のアプリロジック・描画処理は Go に集約し、JavaScript は**純 Go 代替が存在しない既製ライブラリ**（mermaid.js・htmx）と、ごく少量の DOM グルーに限定する。

## 3. 対象プラットフォーム

- **初回対象: Windows**
- **最終目標: Windows / macOS / Linux（デスクトップ3種）**
- モバイル（iPhone / Android）は**対象外**（Wails 採用に伴いモバイル不可。この前提で確定）
- 配布形態: ビルド成果物をまとめた簡素なアーカイブ（Windows: `.exe` を ZIP／macOS: `.app` を ZIP／Linux: AppImage〔仮決め〕）。インストーラ形式は採らない

## 4. 技術選定の結論（2026-06-28 改訂）

| レイヤー | 採用技術 | 備考 |
|----------|----------|------|
| **実装基盤** | **Wails v2**（安定版） | Go バックエンド ＋ OS 標準 WebView。v3 は alpha のため不採用 |
| **中核言語** | **Go** | ファイル I/O・状態管理に加え、**Markdown 描画・サニタイズ・ハイライト・UI（HTML）生成まで担う厚い層** |
| WebView | OS 標準 | Win=WebView2 / macOS=WKWebView / Linux=WebKitGTK |
| **UI 生成** | **Go の `html/template` によるサーバサイド HTML 生成 ＋ htmx** | 操作のインタラクティブ化を htmx（約14KB）が担当。Svelte/TS は**廃止** |
| **Markdown 解析** | **goldmark**（純 Go） | CommonMark 準拠＋GFM（表・打消し線・自動リンク・タスクリスト）・脚注。markdown-it から置換 |
| **コードハイライト** | **chroma**（純 Go） | ソース→色付け HTML。highlight.js から置換 |
| **HTML サニタイズ** | **bluemonday**（純 Go） | allowlist 方式。Go 生成 HTML 前提。DOMPurify から置換 |
| **図表** | **mermaid.js**（WebView 内） | 純 Go/ネイティブ代替が無いため JS を維持（§5.2）。将来 mmdr 検討（§6） |
| **編集モード** | **透明 textarea オーバーレイ ＋ chroma ハイライト** | 編集を担う透明 `<textarea>` を、chroma(Go) 出力の色付け層に重ねる。CodeMirror 6 は**廃止**（§5.3） |
| **PDF 出力** | WebView の印刷 → PDF | 専用ライブラリ不要 |
| **スタイル変更** | CSS 変数（Go がスタイル定義→CSS を出力） | テーマ切替 |
| **マルチタブ** | Go 側の状態 ＋ htmx で管理 | 1ウィンドウ内のタブバー |

## 5. なぜ Go 中心（サーバサイド描画）か

### 5.1 「ほぼ Go」化の3案比較

「ほぼ Go」を実現する構成を3案で比較した（コード量は概算。既製ライブラリは行数に含めない。Go 率は `*.go` のみ／`html/template` を Go 側に含めた値の二通り）。

| 案 | 構成 | 想定 Go 率 | MUST 適合 | 判定 |
|----|------|-----------|-----------|------|
| **A（採用）** | Wails v2 ＋ Go-SSR（goldmark/chroma/bluemonday）＋ htmx ＋ mermaid.js | **約 62〜84%** | ✅ 全て満たす | **採用**。品質・高速起動を保ち、自作 JS を数百行のグルーまで圧縮。Svelte/TS の自作コードを一掃 |
| B | Go→WASM フロント（go-app / vugu / vecty） | 約 91% | ⚠ 抵触リスク | Go 率は高いが **WASM が数 MB で「高速起動」MUST と衝突**。vugu/vecty は実験段階で成熟度が低い |
| C | 純 Go GUI（Fyne / Gio）、WebView 廃止 | 約 98〜100% | ❌ 抵触 | Go 率最大・JS ゼロだが **Markdown 描画品質が低く「綺麗にレンダリング」MUST に抵触**、PDF も困難、mermaid は外部レンダラ必須 |

参考: 改訂前（Wails+Svelte/TS）の Go 率は約 19%（自作 JS/TS が約 75%）。

**結論**: 最大 Go 率を狙う B/C は MUST（高速起動／綺麗な閲覧）と引き換えになる。**A が、品質・起動を維持しつつ現実的に最大限 Go 化できる唯一の案**。Go 率は `*.go` のみで約 62%、Go が生成する `html/template` を含めれば約 84%。

### 5.2 mermaid の制約（JS を完全には消せない）

- **mermaid を純 Go で描画する手段は存在しない**（純 Go の go-mermaid 等はパーサ/検証のみで描画不可）。
- Go 内の JS エンジン（goja / v8go）は **DOM を持たない**ため、mermaid.js を Go プロセス内で走らせることも不可（mermaid は DOM/SVG レイアウトに依存）。
- ヘッドレスブラウザ同梱は「単体アプリ・軽量・高速起動（MUST）」に反する。
- → **mermaid.js を WebView 上で動かす方式を維持**する。これは Wails 採用の根拠そのもの。**この一点で JS は完全には排除できない**ため、「ほぼ Go」の現実的な到達点として許容する。

### 5.3 各ライブラリ置換の要点

- **goldmark**: CommonMark 準拠＋GFM・脚注を備え、markdown-it の機能を代替できる。mermaid コードブロックは `<pre class="mermaid">` として出力し、WebView 側の mermaid.js が描画する。
- **chroma**: 純 Go の構文ハイライタ。閲覧時は Go が色付け HTML を生成。編集時も同じ chroma をオーバーレイ層の色付けに使う。
- **bluemonday**: 純 Go の allowlist サニタイザ。HTML を Go 側で生成・無害化してから WebView に渡す。
- **htmx**: Go 生成 HTML を最小の JS で操作可能にするライブラリ（自作ロジックではない）。タブ切替・モード切替・設定変更等を Go 主導にできる。
- **編集オーバーレイ**: 透明 `<textarea>`（編集・キャレットを担当）を、chroma 出力の `<pre><code>` 層に重ねる定番手法。ハイライト本体は Go、JS はスクロール同期・層差し替えの数十行のグルーのみ。CodeMirror（JS の大きな塊）を排除できる。

詳細なライブラリ比較は [`代替技術調査.md`](代替技術調査.md) を参照。

## 6. mermaid 描画の将来オプション（mmdr）— 情報のみ・採用未確定

将来 JS を更に減らす選択肢として **mmdr（mermaid-rs-renderer）** を記録しておく（**現時点では採用しない**）。

- **概要**: Rust 製のネイティブ mermaid レンダラ。mermaid 構文を**ブラウザ/Node 無しで SVG へ描画**する（mermaid-cli の 500〜1000倍速）。Rust ライブラリまたは CLI として利用可能。
- **これを使うと**: Go から mmdr を呼んで mermaid を**サーバサイドで SVG 化**でき、WebView から **mermaid.js を排除**できる（残る JS は htmx ＋ 編集グルーのみ）。
- **採用を未確定とする理由**:
  - **Rust サイドカー依存**（外部バイナリ同梱 or FFI）が増える。「ほぼ Go」だが純 Go ではない。
  - **v0.2 系と若く、mermaid の全ダイアグラム種別を網羅していない可能性**。mermaid 対応は MUST のため、未対応図種があると致命的。
  - mermaid.js（公式・完全実装）に対する**描画品質・互換性の実地検証が前提**。
- **判断**: 将来、本アプリで使う図種のカバレッジを実地検証できた場合に限り、M1（mermaid.js）→ M2（mmdr）への移行を検討する。それまでは **mermaid.js を採用**する。

## 7. 不採用とした主な候補

| 候補 | 不採用理由 |
|------|-----------|
| **Go→WASM フロント**（go-app / vugu / vecty） | Go 率は高いが、WASM バイナリが数 MB で「起動が素早い（MUST）」と衝突。vugu/vecty は実験段階で成熟度が低くリスク大（§5.1 案B） |
| **純 Go GUI**（Fyne / Gio） | WebView を持たず mermaid 描画が困難。Markdown の綺麗な描画・PDF 出力でも不利で MUST に抵触（§5.1 案C） |
| Electron | 「起動が素早い（MUST）」と衝突（重い・大きい） |
| Tauri v2 (Rust) | Wails と同等の有力候補だったが、言語の好み（Go）を優先。モバイル対応の道は閉じるが対象外のため許容 |
| Wails v3 | 2026 年時点で alpha。安定版の v2 を採用 |

## 8. 既存アプリ調査（参考）

「軽量な Markdown ビュワー」観点での既存アプリ調査の結論:

- 機能の足し算では **Typora**（有償・クローズド・WYSIWYG 主体）がほぼ全要件を満たす。
- ただし「**軽量ネイティブ ＋ マルチタブ ＋ 閲覧/編集の明確な2モード切替**」を兼ね備えたアプリは既存に乏しく、ここが本アプリの差別化ポイント。
- 参考実装: MerMark Editor（Tauri 製）, mdr（Rust 製ビュワー）, VNote（Qt 製）。
